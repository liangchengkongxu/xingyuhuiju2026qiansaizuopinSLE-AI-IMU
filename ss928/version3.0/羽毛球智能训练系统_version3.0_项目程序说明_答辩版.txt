================================================================================
一、项目总体定位
================================================================================

version3.0 是「羽毛球智能训练系统」的 SS928 开发板端主作品代码，负责：

  1）Qt 触摸面板人机交互（linuxfb + GFBG 叠层）
  2）摄像头预览与 NPU 视觉 AI（动作分类、人体框、Pose 骨骼、击球回放）
  3）星闪（SLE）接收拍柄九轴 IMU，做挥拍检测与 1D CNN 击球分类
  4）三种训练模式：单人练习、对打/比赛、班级同训

与上一版 version2.0 的关系：
  - version2.0：面板 + 星闪基础，无完整 attach AI
  - version3.0：在 2.0 基础上增加分类/检测/Pose/回放/IMU CNN 与完整业务 UI

三、运行时进程架构（板上实际跑哪些程序）
================================================================================

板端启动入口：scripts/run.sh → 启动 vo_gfbg_init

进程关系（默认 attach 方案）：

  vo_gfbg_init（C 启动器）
    │
    ├─ 调用 camera_pipe：初始化 MPP（VB/SYS/VI/VPSS/VO），开 HDMI 预览
    │
    ├─ fork sample_vio_ai：attach 到已有 VPSS，做 NPU 推理与 Region 画框
    │
    ├─ 配置 GFBG（图形帧缓冲）
    │
    └─ fork widget_panel（Qt）：触摸 UI，叠在 GFBG 上

配合常驻/按需进程：
  - sle_imu_bridge.sh + sle_seek_print_all：星闪扫描，写 /tmp/sle_imu_lines
  - systemd：widget_ui.service 可开机自启整套面板

进程间通信（文件/管道，无复杂中间件）：
  /tmp/.widget_yolo_action     AI → Qt：当前动作分类结果
  /tmp/.widget_cam_vo          Qt → vo_gfbg_init：摄像头小窗位置 show/x/y/w/h
  /tmp/.widget_replay_session  Qt ↔ AI：击球回放 session
  /tmp/sle_imu_lines           星闪桥 → Qt：IMU @ 行
  /tmp/sample_vio_ai.log       AI 调试日志
  /tmp/widget_imu.log          IMU/CNN 击球日志
  /tmp/widget_panel.log        Qt 面板日志

--------------------------------------------------------------------------
4.1 启动器 vo_gfbg_init（src/vo_gfbg_init.c → bin/vo_gfbg_init）
--------------------------------------------------------------------------

职责：
  - 统一拉起「摄像头管道 + AI attach + GFBG + Qt 面板」
  - 监听 /tmp/.widget_cam_vo，动态调整 VO 显示窗口（摄像头页显示小窗，离开则隐藏）
  - 进入摄像头页时再拉起 sample_vio_ai（按需启动，节省资源）
  - 支持环境变量切换 AI 后端：
      WIDGET_AI_BACKEND=attach   （默认，与 Qt 共存）
      WIDGET_AI_BACKEND=modelzoo （原厂 sample_yolov8，与面板摄像头互斥）

实现要点：
  - C 语言，交叉编译 aarch64
  - 信号处理后有序退出：停 AI → 停面板 → camera_pipe_shutdown
  - 编译脚本：scripts/build_vo_gfbg_init.sh


--------------------------------------------------------------------------
4.2 摄像头管道 camera_pipe（src/camera_pipe.c / .h）
--------------------------------------------------------------------------

职责：封装海思 MPP 媒体通路，供 vo_gfbg_init 在启动 Qt 前调用。

数据流：
  VI(OS08A20) → VPSS grp0
    ├─ ch0  3840×2160   HDMI/VO 全屏预览 + Region 人体框叠加
    ├─ ch1   224×224    动作分类模型输入（NV12）
    ├─ ch2   640×640    人体检测 / Pose 推理输入（NV12）
    └─ ch3   960×540    击球回放专用（不 bind VO，避免抢 ch0 预览）

关键 API：
  camera_pipe_mpp_prepare()     VB + SYS + VI/VPSS 模式初始化
  camera_pipe_preview_start()   启 VI→VPSS→VO，bind 预览
  camera_pipe_vo_set_window()   调整 VO 矩形（与 Qt 透明区对齐）
  camera_pipe_shutdown()        逆序释放

MIPI 插座：默认 WIDGET_CAM_MIPI=1（与部分板厂 FPC 接线一致），可环境变量覆盖。


--------------------------------------------------------------------------
4.3 NPU 视觉 AI：sample_vio_ai（ai/ 目录）
--------------------------------------------------------------------------

可执行文件：bin/sample_vio_ai（板端 /opt/widget_ui/bin/sample_vio_ai）
编译脚本：scripts/build_vio_ai.sh

源码拆分：
  sample_vio_ai.c      主循环：attach VPSS、加载 ACL 模型、分类/检测后处理、写状态文件
  vio_ai_pose.c        Pose 骨骼：加载 best_pose_aipp.om、关键点解码、RGN/软件画线
  vio_ai_hit_replay.c  击球回放：环形缓冲、前后 2s 采集、导出 PPM、触发骨骼 stamp
  vio_ai_yuv.c         YUV 缩放、letterbox、画框画线等图像工具
  vio_ai_env.c         环境变量读取
  vio_ai_internal.h    模块间共享类型与 API

双视觉模型分工：

  【动作分类】
    模型：/opt/widget_ui/models/best_aipp_fix.om
    输入：VPSS ch1，224×224 NV12
    输出：5 类 logits → softmax
    类别：放网 / 高远 / 平抽 / 杀球 / 挑球
    结果写入 /tmp/.widget_yolo_action，供 Qt 显示

  【人体框检测】
    模型：/opt/yolov8n.om（板端路径）
    输入：VPSS ch2，640×640 NV12
    输出：1×10×8400（10 通道 = 4 框坐标 + 5 类）
    说明：当前 OM 为动作检测形态，人体框按「大框 + 合适宽高比」几何优选，
          不是 COCO 80 类 person ID；Region 一次性映射到 ch0 3840×2160 预览

  【Pose 骨骼】
    模型：/opt/widget_ui/models/best_pose_aipp.om
    实时预览：默认 WIDGET_POSE_RGN=0（不画硬件骨骼，减轻与 ch0 抢资源）
    击球回放：pose_stamp_on_replay_nv12() 在导出 PPM 上软件画骨架（BT.601）

  【击球回放】
    通道：VPSS ch3（960×540 @ 约 20fps）
    每次击球：前后各约 2s → /opt/widget_ui/replays/{session}/hit_{N}/
    板端无 ffmpeg：用 PPM 帧序列 + Qt FrameReplayWidget 播放
    策略：前 3 次击球立即导出带骨骼 PPM；第 4 次起先存 raw，进详情页再渲染

4.6 星闪与 IMU 子系统
--------------------------------------------------------------------------

（1）底层扫描工具（ws73/）

  sle_seek_print_all/
    - C 程序：加载星闪协议栈后扫描 ADV / Scan Response
    - 解析厂商域 0xFF 内 22 字节二进制：EB 1A 02 + 九轴数据
    - 统一转为 ASCII「@」行，供上层使用
    - 蓝牙旧固件仍可走 ASCII Notify 路径

  sle_imu_bridge.sh
    - 持续扫描，将 IMU 行写入 /tmp/sle_imu_lines

  sle_seek_bridge.sh
    - 短时扫描，给多人/绑拍柄设备列表用

  ws73_common.sh / ws73_boot_prep.sh
    - 上电、加载 plat_soc.ko / sle_soc.ko、与 WiFi 互斥处理

  sle_announce_cmd / sle_connect_imu
    - 调试或历史路径（连接收 Notify 已弃用为主路径）

（2）Qt 服务层

  sle_seek_service.cpp
    - 封装设备扫描结果，供绑拍柄 UI 列表

  sle_imu_service.cpp
    - 读 /tmp/sle_imu_lines，按 MAC 分发到各拍柄检测器
    - 合并 CNN 触发与规则 FSM，写 /tmp/widget_imu.log
    - 向页面发「击球事件」信号

  sle_measure_cmd.cpp
    - 测距/测量相关命令封装（辅助）

（3）挥拍检测与分类

  imu_swing_detector.cpp
    - 解析 @ 行：@t,Ax,Ay,Az,Gx,Gy,R,P,M
    - 填充 8 通道：ax,ay,az,gx,gy,pitch,roll,M
    - 规则 FSM：动态加速度 / 角速度阈值判定「一次挥拍」
    - 大幅波动 burst：dyn/gyro 超阈可单独作为触发源

  imu_cnn_classifier.cpp + imucnn/
    - 以 M 峰值为中心取 24 帧窗口 → 标准化 → NPU 推理
    - 输入 shape：1×8×24
    - 输出 6 类：高远 / 平抽 / 挑球 / 放网 / 发球 / 杀球
    - 模型：/opt/widget_ui/models/badminton_npu.om
    - 推理库：imucnn/badminton_npu_infer.c（ACL）

  触发模式 WIDGET_IMU_HIT_MODE：
    cnn   — 仅 CNN 硬/软触发
    rule  — 仅规则 FSM + classifySwing
    both  — 默认：CNN 优先，未命中时规则补漏（适合放网、挑球）

  日志含义：
    hit(cnn)  — CNN 路径判定并分类
    hit(rule) — 规则判定挥拍后再分类
    （前缀表示「谁判定算击球」，类型一律由 CNN argmax）

================================================================================
五、三种业务模式如何实现
================================================================================

--------------------------------------------------------------------------
5.1 单人练习
--------------------------------------------------------------------------

页面流：
  首页 → 单人 → 绑拍柄(page14) → 选技能(page2) → 要领/视频(page3)
       → 训练中(page4) → 总结 → 动作详情(可回放)

击球计数：三路 OR（去重冷却约 950ms）
  1）摄像头稳定识别（分类置信度足够高）
  2）摄像头挥拍事件（可选 YOLO swing）
  3）拍柄 IMU（CNN 或规则/burst）

注意：单人练习 UI 上的「击球类型」固定为当前所选技能（m_currentSkill），
      不跟 CNN/YOLO 实时分类结果跳变；便于「练指定动作」的评分与统计。

回放：训练中触发 → sample_vio_ai 从 ch3 采帧 → 总结/详情页用 PPM 序列播放，并叠骨骼。


--------------------------------------------------------------------------
5.2 对打 / 比赛
--------------------------------------------------------------------------

页面流：
  比赛入口 → MatchSetup 绑最多 4 台拍柄 → MatchRunning 双栏计分 → MatchReport

实现特点：
  - 以 IMU 触发为主计「一次击球」
  - 视觉分类可更新击球类型显示
  - 多拍柄按 MAC 独立统计，避免串计数


--------------------------------------------------------------------------
5.3 班级同训
--------------------------------------------------------------------------

特点：
  - 无摄像头预览（减轻 MPP 负载）
  - 多学员卡片：每人绑一台拍柄（MAC）
  - 击球类型完全依赖 IMU 1D CNN（6 类）
  - 点击某次记录进入 ClassHitDetailPage（大字号类型 + AI 文字建议）

数据路径：
  拍柄广播 → sle_seek_print_all → /tmp/sle_imu_lines
    → SleImuService → 每 MAC 独立 detector
    → CNN/规则触发 → ClassTrainPage 卡片 +1


================================================================================
六、关键技术实现要点（答辩可展开）
================================================================================

1）GFBG + Qt 叠层
   - 视频走硬件 VO，UI 走 GFBG/linuxfb
   - 摄像头区域 Qt 透明，由 VO 穿出；位置通过 /tmp/.widget_cam_vo 同步

2）AI attach，而非独占 VI
   - vo_gfbg_init 先建好 VPSS，sample_vio_ai 只 attach 取帧推理
   - 可与 Qt 面板同时存在（区别于 modelzoo 独占模式）

3）多通道 VPSS 分工
   - 预览 / 分类 / 检测 / 回放通道分离，回放切勿占用 ch0（曾导致预览饿死、MPP 崩溃）

4）星闪广播二进制协议
   - 22 字节 EB 1A 02，ADV 与 ScanRsp 双份携带
   - 主控统一转 @ 行，与蓝牙 ASCII 路径兼容

5）IMU 8 通道与训练对齐
   - ch5 必须为 pitch（曾误填 0 导致分类偏向某一类）
   - CNN 硬触发 + 软触发，兼顾杀球与放网/挑球

6）Pose 策略
   - 实时默认不画 RGN；回放软件 stamp，保证训练流畅与回放可看骨骼
