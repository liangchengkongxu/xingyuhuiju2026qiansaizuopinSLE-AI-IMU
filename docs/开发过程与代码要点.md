# 智能羽毛球训练系统 · 开发过程与代码要点

星羽汇聚 2026 嵌赛全栈开发记录：BS20 拍柄固件 + SS928 主控端侧智能。便于答辩、交接与 Cursor Agent 延续开发。

> **当前协议（2026-07，499a3735）**：星闪 **22B `EB 1A 02` 二进制**，ADV + Scan Response 双份 0xFF，主控 **MAC + uptime_ms 去重**，有效采样 **10Hz**。  
> BLE Notify 仍为 ASCII `@` 行。

## 3. 开发里程碑

### 阶段 1：拍柄传感基础（BS20）

- MPU-9250 软件 I2C（MGPIO15/16），UART0 115200 调试
- 上电陀螺零偏校准：20 样本 × 50ms，约 1s
- Mahony AHRS → roll/pitch，100ms 主循环

### 阶段 2：无线上报通路

| 通路 | 实现 | 用途 |
|------|------|------|
| BLE Notify | `paibing_ble_uart.c` ASCII 行 | 调试、单设备验证 |
| SLE 扫播 | `paibing_sle_server_adv.c` | 量产、多设备 |

设备名 `paibing_imu`，MAC 前缀 `cc:ad:c9:00:22:xx`。

### 阶段 3：星闪协议试错（关键转折）

| 尝试 | 问题 | 结论 |
|------|------|------|
| ADV 内长 ASCII (~50B) | 空口时间长，2m 外丢包 | 弃用 |
| 22B `EB 1A 02` 二进制 | 主控按 ASCII 解 → G/R 乱数；按二进制解 → 与 UI 单位不一致 | **回退** |
| 热更新 `set_announce_data` | 播着时不刷新，uptime 卡死 | 必须 **stop→set→start** |
| 180ms 限流 restart | 数据刷新慢，重复帧过久 | 恢复 100ms |
| 仅 ADV 带数据 | 远距离「有名无数据」 | Scan Response 也带 0xFF |

**定型方案**：ASCII 行（与 BLE 相同），ADV+ScanRsp 双份，100ms 异步 restart，广播间隔 5ms。

### 阶段 4：主控星闪接收（SS928 + WS73）

- `sle_seek_print_all`：SLE TLV 解码，厂商 0xFF 取载荷
- 解析演进：二进制优先 → **ASCII 优先**，二进制仅回退
- `sle_imu_bridge.sh` 守护进程 → `/tmp/sle_imu_lines`
- **去重**：拍柄 5ms 广播 vs 100ms 换帧 → 同 `@uptime` 重复 2～20 次
  - C 层：`imu_dedup_allow(mac, uptime)` in `sle_seek_print_client.c`
  - Qt 层：`m_lastSampleTByMac` in `sle_imu_service.cpp`

### 阶段 5：端侧智能与击球检测

- **1D CNN**：`badminton_npu.om`，8 通道窗口，六类挥拍分类
- **规则 FSM**：`imu_swing_detector.cpp`，dyn/gyro 阈值触发
- **模式**：`WIDGET_IMU_HIT_MODE=both` — CNN 峰值优先，规则补漏 + CNN 分类
- **多源触发**（单人练习）：摄像头稳定识别 / YOLO 挥拍 / IMU 三路 OR，950ms 去重

### 阶段 6：训练 UI 业务

| 模式 | 要点 |
|------|------|
| 单人练习 | 动作教学视频 + IMU/视觉击球计数 |
| 班级模式 | 多学员卡片、IMU 击球统计 |
| 对打模式 | 最多 4 台拍柄绑定、多设备布局、IMU 触发 + 视觉延迟更新 |

入口路由：`main_window.cpp` page 16 `MatchSetupPage` 等。

### 阶段 7：联调优化与工程化

| 事项 | 处理 |
|------|------|
| 击球太灵敏 | 提高 dyn/gyro 阈值、confirm=2、cooldown=700ms |
| IMU 数据「看起来不对」 | 区分 ASCII centi-g vs 二进制 mg；校准帧 G/R/P=0 正常 |
| GitHub 文档化 | 分 `bs20/`、`ss928/`，`reference/` 参考代码 |
| 提交版整理 | `羽毛球智能训练系统_提交版/` 双端 README |

### 阶段 8：Pose 骨骼 RGN + 单人击球回放

| 模块 | 实现 |
|------|------|
| Pose 推理 | VPSS ch2 640×640 → `best_pose_aipp.om`，RGN LINE 叠加 ch0 4K 预览 |
| 骨骼稳定 | snap / 滞后可见性 / bbox 大跳 hard_reset |
| 回放采集 | 击球 → `/tmp/.widget_replay_req` → `sample_vio_ai` 环形缓冲 + post 采集 |
| 回放输出 | ch1 640 源 → **960×540** PPM + `pose_stamp_on_replay_nv12()` 软件画骨骼 |
| UI | `ActionDetailPage` 轮询 `done.flag`；`FrameReplayWidget` 读 `meta.txt` fps |

**关键踩坑**：

| 现象 | 根因 | 修复 |
|------|------|------|
| 第 3 次起回放一直「生成中」 | 离开训练页 `clearReplaySession()` 断采集 | 仅 `goHome()` 清 session |
| 骨骼消失 / sample_vio_ai 崩溃 | 从 **VPSS ch0** 高频抢帧，与 RGN/VO 预览争用 | `WIDGET_REPLAY_VPSS_CHN=-1`，attach ch1 异步 worker |
| 回放比例/颜色异常 | 640 拉伸到错误分辨率 | letterbox + 960×540 固定输出 |

详细文档：[`ss928/version3.0/skill/hit-replay-and-pose-rgn.md`](../ss928/version3.0/skill/hit-replay-and-pose-rgn.md)

## 4. 联调踩坑表（精选）

| 现象 | 根因 | 修复 |
|------|------|------|
| BurnTool 超时 | 误烧 `fota.fwpkg` | 用 `*_all.fwpkg` |
| 扫到名无 IMU | 只看了 Scan Response 设备名 | 解析 0xFF 厂商域 |
| G/R 巨大乱数 | 主控按二进制解 ASCII | ASCII 优先解析 |
| `@1281` 重复十几行 | 5ms 广播 + 无去重 | MAC+uptime 去重 |
| uptime 长期不变 | 拍柄广播未 restart | 查 `paibing_sle_adv_rf_publish` |
| A+420 以为 42g | ASCII 下 A×10=mg | 文档标明单位 |
| 随便动就击球 | 规则阈值过低 | 调 `WIDGET_IMU_RULE_*` |
| RSSI「80多」 | 实际是 -80 dBm | 文档说明 |
| 回放卡「生成中」 | session 被 UI 过早清除 | 仅首页 `goHome()` 清 session |
| 骨骼全灭 | ch0 回放抢帧 | `WIDGET_REPLAY_VPSS_CHN=-1` |

## 6. 当前协议要点

### ASCII 行

```
@10222,A-46,+32,+77,G-119,-248,R+169,P-244,M96\n
```

### 拍柄空口行为

- 采样/换帧：**100ms**
- 广播间隔：**5ms**（重复扫描是正常现象）
- 刷新方式：**异步 stop → set_data → start**

### 主控处理

1. ADV 与 Scan Response 均尝试取 `type=0xFF`
2. sscanf / 正则解析 ASCII
3. 同 MAC + 同 uptime → 丢弃
4. 下游挥拍检测消费约 10Hz 有效帧
