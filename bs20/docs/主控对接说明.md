# 主控（SS928）对接说明

> **固件版本**：2026-07 星闪扫播版（**22B 二进制 EB 1A 02** + 异步重启 + ADV/ScanRsp 双份）  
> **拍柄烧录包**：`paibing_sle_mac01_all.fwpkg` … `mac08_all.fwpkg`  
> **详细二进制格式**：见 [主控对接说明-远距离二进制版.md](./主控对接说明-远距离二进制版.md)

---

## 一、主控必须做的事（Checklist）

- [ ] **解析 22 字节二进制**（头 `EB 1A 02`），**不要**按 ASCII 行 sscanf
- [ ] 从 **ADV 或 Scan Response** 的 `0xFF` 厂商域取载荷（两路内容相同）
- [ ] 按 **uptime_ms（偏移 4～5，uint16 LE）+ MAC** 去重
- [ ] 过滤设备名 `paibing_imu` 或 MAC 前缀 `cc:ad:c9:00:22:xx`
- [ ] 预期 **约 100ms 一条新数据**（10Hz）；同一 uptime 重复多次是正常的
- [ ] RSSI 为 **负数 dBm**（如 -80），不是发射功率

---

## 二、当前固件行为（拍柄侧）

| 项目 | 值 |
|------|-----|
| 模式 | 星闪非连接可扫描广播 |
| 设备名 | `paibing_imu` |
| MAC 前缀 | `cc:ad:c9:00:22:xx` |
| 采样/换帧 | **100ms**（10Hz） |
| 广播间隔 | **10ms**（协议栈最小合法值 `0x50`） |
| 发射功率 | ~6～8 dBm（`sle_customize_max_pwr(8,2)`） |

### 广播包结构

| 包类型 | 内容 |
|--------|------|
| **ADV** | Discovery + Access + **0xFF 厂商域（22B 二进制）** |
| **Scan Response** | TX Power + 设备名 + **0xFF 厂商域（同一份 22B）** |

拍柄每 100ms 更新 IMU 后 **异步停播→换数据→再开播**（热更新无效，必须 restart 才能刷新空口）。

---

## 三、数据格式（22 字节二进制）

完整格式见 [**主控对接说明-远距离二进制版.md**](主控对接说明-远距离二进制版.md)。

```
EB 1A 02 DEV  uptime(2)  ax(2) ay(2) az(2)  gx(2) gy(2)  roll(2) pitch(2)  mag(2)
```

| 偏移 | 字段 | 类型 | 换算 |
|------|------|------|------|
| 0-1 | 厂商 ID | — | `EB 1A` |
| 2 | 协议 | u8 | `0x02` |
| 3 | 设备号 | u8 | 默认 `0x01` |
| 4-5 | uptime | u16 LE | 毫秒低 16 位 |
| 6-11 | ax,ay,az | i16 LE | ×10 → mg |
| 12-15 | gx,gy | i16 LE | dps |
| 16-19 | roll,pitch | i16 LE | ÷10 → 度 |
| 20-21 | magnitude | u16 LE | ÷100 → g |

### 解析步骤

1. 星闪扫描（Seek），拿到 ADV 和/或 Scan Response 原始字节
2. 遍历 AD 结构：`length | type | value...`
3. 找 **`type == 0xFF`**（Manufacturer Specific Data）
4. 校验 `payload[0]==0xEB && payload[1]==0x1A && payload[2]==0x02`
5. 按偏移读 int16/u16（小端）

## 四、主控必改：去重逻辑

因广播间隔 10ms，**同一 uptime 会被扫到多次**，例如：

```
uptime=11099 ...   ← 同一条
uptime=11099 ...   ← 重复，应丢弃
uptime=11398 ...   ← 新帧，处理
```

**推荐做法**（每个 MAC 维护 last_uptime）：

```c
static uint32_t g_last_uptime[40];  /* 按 MAC 索引 */

bool paibing_should_accept(const uint8_t mac[6], uint32_t uptime_ms)
{
    int idx = mac[5] - 1;  /* 或按 MAC 查表 */
    if (idx < 0 || idx >= 40) return true;
    if (uptime_ms == g_last_uptime[idx]) {
        return false;  /* 重复帧，丢弃 */
    }
    g_last_uptime[idx] = uptime_ms;
    return true;
}
```

只把 **uptime 变化** 的帧送给业务层，日志就不会刷屏，且保持 10Hz 有效数据率。

## 五、参考解析代码（C）

```c
typedef struct {
    uint32_t uptime_ms;
    int32_t ax_mg, ay_mg, az_mg;
    int32_t gx_dps, gy_dps;
    float roll_deg, pitch_deg;
    float magnitude_g;
} paibing_imu_sample_t;

int paibing_parse_mfg22(const uint8_t *p, paibing_imu_sample_t *out)
{
    if (p[0] != 0xEB || p[1] != 0x1A || p[2] != 0x02) return -1;
    out->uptime_ms = (uint32_t)(p[4] | (p[5] << 8));
    out->ax_mg = (int16_t)(p[6] | (p[7] << 8)) * 10;
    out->ay_mg = (int16_t)(p[8] | (p[9] << 8)) * 10;
    out->az_mg = (int16_t)(p[10] | (p[11] << 8)) * 10;
    out->gx_dps = (int16_t)(p[12] | (p[13] << 8));
    out->gy_dps = (int16_t)(p[14] | (p[15] << 8));
    out->roll_deg = (int16_t)(p[16] | (p[17] << 8)) / 10.0f;
    out->pitch_deg = (int16_t)(p[18] | (p[19] << 8)) / 10.0f;
    out->magnitude_g = (p[20] | (p[21] << 8)) / 100.0f;
    return 0;
}

const uint8_t *paibing_find_mfg_payload(const uint8_t *ad, uint16_t ad_len,
    uint16_t *out_len)
{
    uint16_t idx = 0;
    while (idx < ad_len) {
        uint8_t field_len = ad[idx];
        if (field_len == 0 || idx + 1 + field_len > ad_len) break;
        if (ad[idx + 1] == 0xFF) {
            *out_len = (uint16_t)(field_len - 1);
            return &ad[idx + 2];
        }
        idx = (uint16_t)(idx + 1 + field_len);
    }
    return NULL;
}
```

### 完整处理流程伪代码

```c
void on_sle_scan_result(const sle_seek_result_t *r)
{
    paibing_imu_sample_t sample;
    const uint8_t *payload;
    uint16_t plen;

    if (!mac_prefix_match(r->addr, 0xCC, 0xAD, 0xC9, 0x00, 0x22)) return;

    payload = paibing_find_mfg_payload(r->adv_data, r->adv_len, &plen);
    if (payload == NULL) {
        payload = paibing_find_mfg_payload(r->scan_rsp, r->scan_rsp_len, &plen);
    }
    if (payload == NULL || plen < 22) return;

    if (paibing_parse_mfg22(payload, &sample) != 0) return;
    if (!paibing_should_accept(r->addr, sample.uptime_ms)) return;

    paibing_on_imu_sample(r->addr, &sample);
}
```

## 六、与旧版差异（请勿再用旧逻辑）

| 项目 | 旧版（已废弃） | **当前固件** |
|------|----------------|-------------|
| 载荷格式 | ASCII 行 ~50B | **22B 二进制** `EB 1A 02` |
| 数据位置 | 仅 ADV | **ADV + Scan Response 均有** |
| 刷新方式 | 热更新 `set_announce_data` | **100ms 异步 restart** |
| 广播间隔 | 0x28（5ms） | **0x50（10ms）** |
| 主控解析 | sscanf ASCII 行 | **按偏移读 int16** |

若主控仍按 ASCII 解析二进制载荷，会出现 **数值错乱、G/R 异常大** 等问题。

---

## 七、蓝牙版（参考，非当前扫播路径）

蓝牙版仍为连接后 **GATT Notify**，格式与上表 ASCII 行相同，无需从广播解析。

---

## 八、RSSI 与距离

- 显示「80 多」通常是 **-80 dBm**（弱信号）
- 2m 外丢包时：优先保证去重逻辑正确，再优化扫描占空比
- 扫到名字 ≠ 收到 IMU：名字在 Scan Response，数据在 0xFF 域

---

## 九、多设备 MAC

| 末字节 | MAC |
|--------|-----|
| 01～08 | `cc:ad:c9:00:22:01` … `:08` |

按 MAC 末字节或完整 MAC 区分拍柄，**去重表按 MAC 独立维护**。
