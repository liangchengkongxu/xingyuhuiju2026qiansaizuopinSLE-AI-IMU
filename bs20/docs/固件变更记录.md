# BS20 星闪固件变更记录（给主控同事）

## 2026-07 当前量产行为（499a3735 · 远距离二进制版）

### 拍柄固件做了什么

1. **星闪非连接扫播**，设备名 `paibing_imu`
2. **载荷格式**：**22 字节二进制** `EB 1A 02`（0xFF 厂商域），**不是** ASCII 行
3. **ADV 和 Scan Response 都带** 同一份二进制（0xFF 厂商域）
4. **100ms 更新一帧** IMU；每帧通过 **异步 stop → set_data → start** 刷新空口
5. 广播间隔 **10ms**（125us×0x50）
6. 功率：`sle_customize_max_pwr(8,2)`，约 6～8 dBm

### 主控必须配合修改

| # | 事项 | 原因 |
|---|------|------|
| 1 | 解析 **22B `EB 1A 02`**，勿 sscanf ASCII | ASCII 解析二进制 → G/R 乱数 |
| 2 | **ADV 和 Scan Response 都尝试** 取 0xFF | 双份提高收包率 |
| 3 | 按 **uptime_ms（偏移 4～5）去重** | 10ms 广播导致同一帧重复 |
| 4 | 预期 **10Hz 有效样本** | 不是每条扫描日志都是新数据 |
| 5 | 不要用 `fota.fwpkg` 烧录拍柄 | 仅 `*_all.fwpkg` |

### 协议演进

| 阶段 | 方案 | 结果 |
|------|------|------|
| 早期 | ADV 内长 ASCII (~50B) | 2m 外丢包 |
| 中期 | 22B `EB 1A 02` 二进制 | 7m+ 收包好；主控需按字节解析 |
| 短暂 | 回 ASCII 双份 | 5m 内可用，7m+ 收包率下降 |
| **当前** | **再改回 22B 二进制** | 7m+ 收包率优先 |

### 曾试验但已回退的方案

| 方案 | 问题 |
|------|------|
| 热更新 `set_announce_data` | 空口不刷新，uptime 卡死 |
| 180ms 限流 restart | 数据刷新变慢 |
| 仅 ADV 无 ScanRsp 数据 | 远距离「有名无数据」 |
