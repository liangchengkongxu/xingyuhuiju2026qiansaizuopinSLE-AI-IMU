# BS20 星闪固件变更记录（给主控同事）

## 2026-07 当前量产行为（远距离二进制版）

### 拍柄固件做了什么

1. **星闪非连接扫播**，设备名 `paibing_imu`
2. **载荷格式**：**22 字节二进制** `EB 1A 02`，**不是** ASCII 行
3. **ADV 和 Scan Response 都带** 同一份 22B（0xFF 厂商域）
4. **100ms 更新一帧** IMU；每帧通过 **异步 stop → disable 回调里 commit+start** 刷新空口
5. 广播间隔 **10ms**（125us×0x50，协议栈最小合法值）
6. 功率：`sle_customize_max_pwr(8,2)` + `announce_tx_power=20`，约 6～8 dBm

### 主控必须配合修改

| # | 事项 | 原因 |
|---|------|------|
| 1 | 改解析为 **22B 二进制** `EB 1A 02` | ASCII 解析会导致数值错乱（如 A+980） |
| 2 | **ADV 和 Scan Response 都尝试** 取 0xFF | 双份提高收包率 |
| 3 | 按 **uptime（偏移 4～5）+ MAC** 去重 | 10ms 广播导致同一帧重复多次 |
| 4 | 预期 **10Hz 有效样本** | 不是每条扫描日志都是新数据 |
| 5 | 不要用 `fota.fwpkg` 烧录拍柄 | 仅 `*_all.fwpkg` |

详细格式与 C 代码见 [`主控对接说明-远距离二进制版.md`](主控对接说明-远距离二进制版.md)。

### 曾试验但已回退的方案

| 方案 | 问题 |
|------|------|
| ASCII ~50B 扫播 | 5m 尚可，7m+ CRC 失败率高 |
| 热更新 `set_announce_data` | 空口不刷新，uptime 卡死 |
| 同步 stop→sleep→start | 0x8000600a 状态冲突 |
| 180ms 限流 restart | 数据刷新变慢，同一帧重复过久 |
| 间隔 0x28（5ms） | 可能低于协议栈最小值 |
| 仅 ADV 无 ScanRsp 数据 | 远距离「有名无数据」 |
