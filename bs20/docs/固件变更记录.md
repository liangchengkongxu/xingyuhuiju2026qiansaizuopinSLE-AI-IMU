# BS20 星闪固件变更记录（给主控同事）

## 2026-07-09 回退说明（实测）

22B 二进制 + 0x50(10ms) 间隔版本**丢包率未改善、不如 ASCII+0x28 版**。

根因：主控扫描窗口内「能扫到的包数」比单包 CRC 更关键；0x50 使发包密度约为 0x28 的一半。

当前恢复：
- **ASCII 行** + **0x28(5ms)** + **同步 stop→start**（缩短停播空窗）
- 主控继续 **ASCII 解析 + @uptime 去重**

## 2026-07 当前量产行为

### 拍柄固件做了什么

1. **星闪非连接扫播**，设备名 `paibing_imu`
2. **载荷格式**：ASCII 行（与 BLE Notify 一致）
3. **ADV 和 Scan Response 都带** 同一份 ASCII（0xFF 厂商域）
4. **100ms 更新一帧** IMU；每帧 **同步 stop → 等 disable → commit+start**
5. 广播间隔 **5ms**（125us×0x28）
6. 功率：`sle_customize_max_pwr(8,2)` + `announce_tx_power=8`，约 6～8 dBm

### 主控必须配合修改

| # | 事项 | 原因 |
|---|------|------|
| 1 | 解析 **ASCII 行** `@ms,A...,G...,R...,P...,M...\n` | 二进制解析会导致数值错乱 |
| 2 | **ADV 和 Scan Response 都尝试** 取 0xFF | 双份提高收包率 |
| 3 | 按 **`@uptime_ms` + MAC** 去重 | 5ms 广播导致同一帧重复多次 |
| 4 | 预期 **10Hz 有效样本** | 不是每条扫描日志都是新数据 |
| 5 | 不要用 `fota.fwpkg` 烧录拍柄 | 仅 `*_all.fwpkg` |

详细格式见 [`主控对接说明.md`](主控对接说明.md)。

### 曾试验但已回退的方案

| 方案 | 问题 |
|------|------|
| 22B 二进制 + 0x50 | 实测丢包率不如 ASCII+0x28 |
| 热更新 `set_announce_data` | 空口不刷新，uptime 卡死 |
| 同步 stop→sleep→start | 0x8000600a 状态冲突 |
| 180ms 限流 restart | 数据刷新变慢，同一帧重复过久 |
| 仅 ADV 无 ScanRsp 数据 | 远距离「有名无数据」 |
