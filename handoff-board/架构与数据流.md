# 架构与数据流

## 系统分工

```
┌─────────────┐     HTTP POST      ┌──────────────────┐     HTTP GET      ┌─────────────┐
│  Hi3403     │ ─────────────────► │ FastAPI 阿里云    │ ◄──────────────── │ Android App │
│  端侧 AI    │  device/ingest/*   │ 47.107.120.9     │  matches/drills  │  星羽汇聚    │
└─────────────┘                    └──────────────────┘                   └─────────────┘
       │                                      │
       │ 可选                                 │ 班级/球友圈/图片
       ▼                                      ▼
┌─────────────┐                         教练看学员数据
│ L610 MQTT   │                         学员/App 看自己的数据
│ ThingsKit   │  （App 不读此通路）
└─────────────┘
```

## 板端职责

| 环节 | 板端做什么 |
|------|-----------|
| 采集 | 摄像头 + Hi3403 跑动作识别 |
| 结构化 | 输出 score、球速、力度、AI 建议、动作类型 |
| 归属用户 | 设备绑定或扫码得到 `user_phone`，写入每条上报 |
| 上报 | L610 4G 发 HTTP POST 到 FastAPI |
| 可选 | 同时 MQTT 发 ThingsKit（实验室调试，非 App 数据源） |

## App 侧如何看到数据

1. 用户用 `user_phone` 在 App 注册登录
2. 板端 POST 时带相同 `user_phone`
3. 后端写入 `matches` / `strokes` / `drill_sessions` 表
4. App 调用 `GET /api/v1/matches`、`GET /api/v1/drills` 展示
5. 若学员加入了教练班级，教练在 **教练端 → 班级 → 学员** 也能看到同一份数据

## 上报时机建议

| 模式 | 建议 |
|------|------|
| 对打/比赛 | 一场结束后 **一次性** POST 整场 + 全部 strokes |
| 单人练习 | **每完成一次** 动作识别 POST 一条 drill |

## 时序（对打）

```
开始比赛 → 逐拍识别（本地缓存）→ 比赛结束 → POST /device/ingest/match
                                              ↓
                                    返回 { ok, matchId }
                                              ↓
                                    用户/App 刷新即可见
```

## 注意事项

- **`user_phone` 必填且准确**，否则数据可能落到错误用户或第一个注册用户
- `device_id` 当前仅记录用途，不参与鉴权
- 图片、社交、班级 **不走板端**，无需板端实现
